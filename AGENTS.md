# 작업 규칙

이 저장소에서는 가능한 한 MCP를 우선 사용한다.

- GitHub 관련 작업은 GitHub MCP를 먼저 사용한다.
- 문서 조회, API 사용법, 스펙 확인은 Docs MCP를 먼저 사용한다.
- 로컬 정보만으로 충분하지 않은데도 추측으로 진행하지 않는다.

API 스펙은 절대 추측하지 않는다.

- 엔드포인트, 요청/응답 필드, 인증 방식, 옵션, 제한사항은 Docs MCP에서 확인한 뒤 사용한다.
- 문서 근거를 찾지 못했으면 구현 전에 먼저 확인 결과를 보고한다.

기본 개발 방법론은 TDD다.

- 구현 전에 테스트 시나리오와 기대 동작을 먼저 정리한다.
- 가능하면 실패하는 테스트를 먼저 만들고, 그 다음 최소 구현으로 통과시킨다.
- 리팩터링은 테스트가 계속 통과하는 상태에서만 진행한다.

PR 전 품질 게이트는 필수다.

- PR을 열기 전 반드시 `lint`와 `test`를 실행한다.
- 두 단계가 모두 성공한 경우에만 `push`한다.
- 하나라도 실패하면 원인을 수정한 뒤 다시 검증한다.

# 역할 분담

작업은 아래 역할로 분리한다.

- `spec-agent`: 요구사항 정리, 범위 정의, 작업 체크리스트 작성
- `backend-agent`: 실제 코드 구현
- `tdd-agent`: 테스트 작성과 테스트 관점 검증
- `quality-agent`: 최종 리뷰, 회귀 위험 점검, 누락된 검증 확인

권장 순서는 다음과 같다.

1. `spec-agent`가 먼저 스펙과 완료 조건을 정리한다.
2. `tdd-agent`가 실패하는 테스트 또는 검증 시나리오를 먼저 만든다.
3. `backend-agent`가 테스트를 통과시키는 방향으로 구현한다.
4. `quality-agent`가 변경사항을 리뷰하고 린트/테스트 실행 여부를 확인한다.

# 에이전트 사용 원칙

이 규칙은 `AGENTS.md`에 두고, 각 역할의 실제 동작 방식과 제약은 `.codex/agents/*.toml`에 정의한다.

- `AGENTS.md`에는 저장소 전체 정책, 역할 분담, 작업 순서를 적는다.
- `.codex/agents/*.toml`에는 각 에이전트의 책임, 허용 작업, 금지 사항, 모델/샌드박스 설정을 적는다.

따라서 위 역할 분담은 `AGENTS.md`에도 적는 것이 맞고, 각 역할의 세부 규칙은 개별 agent 정의에도 함께 적어야 한다.
