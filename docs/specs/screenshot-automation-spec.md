# Screenshot Maker SPEC

## 문서 정보

- 상태: Draft
- 작성일: 2026-03-18
- 작성 목적: 업로드된 앱 스크린샷 이미지를 iPhone/Android 스토어 목업 이미지로 자동 변환하는 풀스택 서비스의 구현 기준 정의

## 배경

현재 작업 흐름은 다음과 같다.

1. Flutter 앱에서 필요한 화면의 스크린샷을 직접 준비한다.
2. 준비한 이미지를 Figma에서 iPhone/Android 목업에 수동으로 끼워 넣는다.
3. 배경, 카피, 정렬을 직접 조정한다.
4. 최종 이미지를 스토어 업로드용으로 정리한다.

이 방식은 반복 작업 비중이 높고, 화면 변경이 생길 때마다 동일한 수동 편집을 다시 해야 한다. 특히 디바이스 프레임 합성, 정렬, 배경 처리 같은 반복 작업이 병목이다.

## 목표

사용자가 업로드한 앱 스크린샷을 기반으로 iPhone/Android 스토어용 목업 이미지를 자동 생성한다.

자동화 결과는 다음 조건을 만족해야 한다.

- Figma 수동 편집 없이도 일정한 스타일의 스토어 목업 이미지를 생성할 수 있어야 한다.
- 하나의 입력 이미지를 기반으로 iPhone/Android 버전을 동시에 만들 수 있어야 한다.
- 제목, 부제, 배경색 같은 공통 스타일을 브라우저에서 조정할 수 있어야 한다.
- 생성 결과를 즉시 미리보기할 수 있어야 한다.

## 문제 정의

현재 병목은 촬영 이후의 후처리와 목업 편집이다.

- 디바이스 프레임 합성과 정렬이 수작업이다.
- 배경과 카피 스타일을 매번 다시 맞춰야 한다.
- 같은 이미지를 iPhone/Android용으로 각각 편집해야 한다.
- 반복 생성 비용이 높다.

## 대상 사용자

- 앱 스크린샷을 스토어용 마케팅 이미지로 자주 변환해야 하는 개발자
- Figma 없이 빠르게 스토어 목업 이미지를 만들고 싶은 1인 또는 소규모 팀

## 제안 범위

### In Scope

- 단일 이미지 업로드 UI
- 제목, 부제, 배경색 입력 UI
- iPhone/Android 기본 목업 템플릿 제공
- 서버에서 이미지 합성 및 PNG 결과 생성
- 브라우저에서 생성 결과 미리보기
- 추후 다운로드/저장 확장을 고려한 API 응답 구조

### Out of Scope

- 앱 화면 자동 캡처
- 스토어 콘솔 자동 업로드
- 다국어 카피 관리 시스템
- 템플릿 에디터
- 사용자 인증

## 핵심 시나리오

1. 사용자가 웹 UI에서 원본 스크린샷을 업로드한다.
2. 사용자가 제목, 부제, 배경색, 템플릿 선택값을 입력한다.
3. 서버가 입력 이미지를 iPhone/Android 템플릿에 맞춰 합성한다.
4. 서버가 PNG 결과를 반환한다.
5. 프런트엔드가 생성된 목업 이미지를 즉시 렌더링한다.

## 기능 요구사항

### FR-1. 이미지 업로드

- 사용자는 단일 이미지 파일을 업로드할 수 있어야 한다.
- 서버는 업로드된 파일이 이미지인지 검증해야 한다.

### FR-2. 템플릿 선택

- 시스템은 iPhone/Android 기본 템플릿을 제공해야 한다.
- 사용자는 어떤 템플릿으로 렌더링할지 선택할 수 있어야 한다.

### FR-3. 이미지 합성

- 서버는 원본 이미지를 디바이스 화면 영역에 맞게 리사이즈/크롭해야 한다.
- 서버는 프레임, 배경, 카피를 합성한 PNG를 생성해야 한다.

### FR-4. 스타일 입력

- 사용자는 제목, 부제, 배경 그라디언트 색상을 입력할 수 있어야 한다.
- 값이 비어 있으면 시스템 기본값을 사용해야 한다.

### FR-5. 결과 미리보기

- 프런트엔드는 생성된 결과를 카드 형태로 보여줘야 한다.
- 각 결과는 어떤 디바이스 목업인지 식별 가능해야 한다.

### FR-6. 재생성

- 사용자는 같은 이미지를 기준으로 스타일 값을 바꿔 다시 생성할 수 있어야 한다.

### FR-7. 실패 감지

- 잘못된 파일 업로드, 렌더링 실패, 지원하지 않는 템플릿을 구분해서 확인할 수 있어야 한다.

## 비기능 요구사항

- 반복 가능성: 동일 입력에 대해 일관된 결과를 생성해야 한다.
- 유지보수성: 새 디바이스 템플릿 추가 시 도메인 계층만 확장하면 되도록 설계해야 한다.
- 확장성: 향후 다운로드 저장, 다국어 카피, 사용자 템플릿으로 확장 가능해야 한다.
- 속도: 업로드 후 수 초 내에 목업 결과를 반환해야 한다.
- 가시성: 실패 이유를 API와 UI에서 파악 가능해야 한다.

## 제약사항

- 백엔드는 Node.js + NestJS, 프런트엔드는 Vite 기반으로 구현한다.
- 구현은 TDD 방식으로 진행해야 한다.
- 이미지 합성은 서버에서 처리한다.
- 현재 범위에서는 정밀한 스토어 규격 대응보다 범용 세로형 목업 생성에 집중한다.

## 설계 원칙

- 디바이스 템플릿 정의와 이미지 합성 로직을 분리한다.
- 프런트엔드는 입력/미리보기에 집중하고, 렌더링 규칙은 백엔드 도메인 계층에 둔다.
- 초기 MVP는 iPhone 1종, Android 1종 템플릿에 집중한다.

## 가정

- 사용자가 직접 준비한 원본 스크린샷을 업로드할 수 있다.
- 프레임 정확도가 완벽한 실물 디바이스 수준이 아니어도 MVP로는 허용한다.

## 오픈 질문

아래 항목은 구현 전에 확정이 필요하다.

- 최종 결과를 서버 저장까지 할지, 현재처럼 즉시 응답만 할지?
- 템플릿 종류를 몇 개까지 확장할지?
- 향후 다운로드 ZIP 패키징이 필요한지?

## 수용 기준

다음 조건을 만족하면 1차 목표를 달성한 것으로 본다.

1. 이미지 파일 업로드가 가능하다.
2. iPhone 1종과 Android 1종 목업을 동시에 생성할 수 있다.
3. 제목, 부제, 배경색이 결과 이미지에 반영된다.
4. 프런트엔드에서 생성 결과를 즉시 확인할 수 있다.
5. 백엔드 도메인 계층에서 템플릿 추가가 가능하다.
6. 잘못된 입력에 대해 명확한 에러를 반환한다.

## 구현 단계 제안

### Phase 1. Validation

- 업로드 UI와 렌더링 API 경계를 정한다.
- 도메인 계층에서 디바이스 템플릿 모델을 정의한다.
- 목업 캔버스와 화면 삽입 규칙을 검증한다.

### Phase 2. MVP

- NestJS API로 업로드 및 목업 생성 기능을 구현한다.
- Vite UI로 업로드 폼과 결과 그리드를 구현한다.
- iPhone/Android 기본 템플릿을 제공한다.
- 기본 그라디언트와 카피 렌더링을 지원한다.

### Phase 3. Expansion

- 다운로드, ZIP 패키징, 저장 기능을 추가한다.
- 다국어 카피와 추가 디바이스를 확장한다.
- 사용자 커스텀 템플릿을 추가한다.

## TDD 관점 테스트 시나리오

구현 전에 아래 검증 시나리오를 우선 테스트 대상으로 삼는다.

1. 디바이스 템플릿 레이아웃 계산이 유효한 화면 영역을 반환한다.
2. 유효한 입력으로 iPhone/Android 목업이 모두 생성된다.
3. 렌더링 결과 PNG 크기가 템플릿 캔버스와 일치한다.
4. 프런트엔드 결과 그리드가 생성 결과를 정확히 표시한다.
5. 잘못된 입력은 에러로 처리된다.

- 텍스트 길이가 길면 레이아웃이 깨질 수 있다.
